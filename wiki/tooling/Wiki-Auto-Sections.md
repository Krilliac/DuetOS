# Wiki Auto-Sections

The wiki holds two kinds of content:

* **Hand-written prose** — design discussion, project history,
  rationale, "why we chose X." Stable, low-churn, edited only when
  the underlying design changes.
* **Inventories derived from source-of-truth artifacts** — the
  list of Win32 thunks, the FixDetector enum, the kernel's shell
  command surface, the marker manifest, the syscall dispatch
  table. These drift the moment a slice lands code in the kernel.
  Keeping them by hand turns "did the slice update the wiki?"
  into a recurring review fight.

The **AUTO-section** convention solves the second class. Each
auto-generated block is bracketed by a pair of HTML comment
markers; a generator regenerates the content between them on
demand and a CI check fails the PR if the wiki on disk doesn't
match what the generator would produce against the current source
tree.

## The marker format

```markdown
<!-- AUTO:<section-id> [key=value [key=value ...]] START -->
... generated content ...
<!-- AUTO:<section-id> END -->
```

* `<section-id>` selects the generator (e.g. `thunks-by-dll`,
  `journal-detectors`, `shell-commands`).
* The `START` marker may carry zero or more `key=value` arguments
  (no spaces in values). Generators read these as parameters; e.g.
  `thunks-by-dll DLL=kernel32.dll` selects rows for one DLL.
* The `END` marker takes no arguments — it just closes the block.
* Anything outside an AUTO block is hand-written and the generator
  never touches it.

A regenerated block always looks like:

```markdown
<!-- AUTO:thunks-by-dll DLL=kernel32.dll START -->
<!-- generated by tools/build/gen-wiki-auto.py — do not edit; regen via `tools/build/gen-wiki-auto.py` -->

| Method | Status | Routed to |
|--------|--------|-----------|
| `ExitProcess` | REAL | `kOffExitProcess` |
| ...
<!-- AUTO:thunks-by-dll DLL=kernel32.dll END -->
```

## How to use

### Authoring an auto block

Pick the generator, write the markers, leave the body empty:

```markdown
## kernel32 — Win32 base API

<!-- AUTO:thunks-by-dll DLL=kernel32.dll START -->
<!-- AUTO:thunks-by-dll DLL=kernel32.dll END -->
```

Then run `tools/build/gen-wiki-auto.py` to fill it in. Commit the
result.

### Regenerating after a slice

```bash
# Regenerate every AUTO block in the wiki/ tree:
python3 tools/build/gen-wiki-auto.py
# Check-only mode (CI):
python3 tools/build/gen-wiki-auto.py --check
```

`--check` exits non-zero if any AUTO block on disk differs from
what the generator would produce. CI uses this to fail PRs that
land code without re-running the generator.

### Adding a new generator

A generator is a Python function in `tools/build/gen-wiki-auto.py`
registered against a `<section-id>`. It receives the kwargs from
the START marker and returns the body text. See the file's module
docstring for the contract.

## Available generators

The current generator set is documented at
[`tools/build/gen-wiki-auto.py`](../../tools/build/gen-wiki-auto.py)
under `_GENERATORS`. Each entry there carries a one-line description
of what it produces; the live registry is the source of truth.

## Design rationale

### Why HTML comments?

GitHub renders Markdown but hides HTML comments. The marker is
visible in source, invisible in the rendered wiki page. The
existing convention `<!-- AUTO:* -->` was already pre-declared in
`CLAUDE.md` ("Prefer upstream changes for auto-generated content
(`<!-- AUTO:* -->` sections) once docs automation is introduced.")
— this slice realises it.

### Why one generator per section ID, not per file?

A wiki page may hold multiple AUTO blocks driven by different
generators. `Win32-Surface-Status.md` carries one
`thunks-by-dll DLL=kernel32.dll` per Win32 DLL section.
`Knowledge-Base-Index.md` could carry one `wiki-toc` to render a
table of contents. Keeping the dispatch per-section means each
block can have its own arguments and its own regeneration cost,
and a generator can serve multiple pages.

### Why not commit-time hooks?

A pre-commit hook that regenerated the wiki on every commit would
work locally but couldn't gate PRs. The `--check` mode in CI is
the gate; local regeneration is opt-in (run before the commit you
want to land cleanly). This matches the existing
`clang-format --dry-run --Werror` pattern for source format.

### What about partial generators (e.g. surfacing prose comments)?

The convention is "between START and END is fully owned by the
generator." A generator that wants to mix human prose with
auto-content embeds the prose itself (e.g. by reading a sibling
file) — it never preserves whatever was between the markers
last time. That keeps `--check` deterministic and avoids the
"the auto block silently shrinks because the human deleted a
line" failure mode.
